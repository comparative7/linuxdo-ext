# LinuxDo Extension - Vibe Coding 开发手册

本项目严格遵循 Vibe Coding 方法论进行开发，绝不将整个项目交由 AI 一次性生成，也绝不在单一且冗长的上下文中开发多个模块。

请严格按照以下五个阶段进行开发。**每个阶段都必须在新开启的 Cursor Composer 或 Chat 窗口中进行，且每完成一个阶段必须进行 Git 提交。**

---

## 🌟 阶段零：全局架构规划（先出图纸，绝不动工）
这是整个项目最关键的一步。此时你只需要和 Agent 探讨，严禁它创建文件。

1. 在 Cursor 中打开一个全新的 Composer (`Ctrl + I`)。
2. 输入终极冷启动指令：
   > “请仔细阅读 `specs/linuxdo_ext_spec.md` 和 `.cursor/rules/chrome_ext.mdc`，以及 `AGENTS.md`。
   > **我们现在处于架构设计阶段。在你写任何代码或创建任何文件之前，请基于需求向我输出一份详细的技术实现计划，必须包含：**
   > 1. 完整的目录树结构规划。
   > 2. `content.js` 和 `background.js` 之间具体通过什么消息格式（Message Schema）进行通信？
   > 3. 你打算如何实现‘避免点到已读帖子和置顶帖’的逻辑？
   > 4. 你觉得这个需求里最难处理的坑是什么？
   > 
   > **注意：没有我的批准，绝对不要创建任何文件。**”
3. **审批**：如果它提议引入多余的库，或者通信逻辑设计得很绕，直接在这个窗口里反驳它，直到你们对计划完全达成共识。
4. **保存图纸**：告诉它：“计划通过，请把这个最终计划总结并写入 `docs/architecture_plan.md` 文件中，然后我们将结束本回合。”
5. 触发 `git-committer` 提交这份图纸，然后**关掉这个 Composer 窗口**。

---

## 🏗️ 阶段一：搭建脚手架（严格按图纸施工）
图纸有了，现在开始铺砖。

1. **新开** Composer 窗口。
2. 输入指令：
   > “请读取刚才确认的 `docs/architecture_plan.md` 和相关规则。
   > 现在开始执行**第一阶段：搭建项目骨架**。请创建 `manifest.json`、基础的 `popup` 文件以及空的 `background.js` 和 `content.js`。
   > 不需要写任何滚动或跳转的业务逻辑。确保它能在 Chrome 中被成功加载即可。”
3. 亲自去 Chrome 扩展管理页验证加载，不报错即可。
4. 触发 `git-committer` 提交代码，**关掉窗口**。

---

## 🏃 阶段二：攻克核心业务 1 —— 页面内拟人化滚动
1. **新开** Composer 窗口。
2. 输入指令：
   > “现在开始编写 `scripts/content.js` 的详情页逻辑。
   > 请严格按照 `architecture_plan.md` 中的设计，在 `linux.do/t/*` 页面实现带随机延迟（3-8秒）的滚动逻辑，滚到底部后向 background 发送完成信号。
   > 先给我看你的实现思路，批准后再写代码。”
3. 去 Chrome 刷新帖子页面，肉眼观察滚动是否足够拟人。
4. 触发 `git-committer` 提交代码，**关掉窗口**。

---

## 🔄 阶段三：攻克核心业务 2 —— 跨页面闭环调度
1. **新开** Composer 窗口。
2. 输入指令：
   > “现在开始编写 `background.js` 以及列表页的调度逻辑。
   > 按照图纸，接收到‘阅读完成’信号后等待 10 秒，控制标签页退回首页，并在主页避开置顶帖寻找下一个未读帖子进入。同时让 Popup 的开关能正确控制这个循环的启动和停止。
   > 先说思路，批准后再动工。”
3. 开启开关，看着它全自动运行一个完整的闭环。
4. 触发 `git-committer` 提交代码，**关掉窗口**。

---

## 🛡️ 阶段四：安全审查与完工
1. 打开普通 Chat 对话框 (`Ctrl + L`)。
2. 触发审查技能：“**调用 security-reviewer，对现在的代码做全面的安全、权限和反爬虫特征审查。**”
3. 根据建议微调代码，进行最后一次 Commit。
4. 执行 `git push` 推送代码，至此核心项目完工。

---

## ➕ 附加指南：如何安全地新增功能？
当项目跑通后，如果你想增加新功能（例如：“增加每日浏览上限”或“劳逸结合机制”），**绝对不能**直接让 Agent 修改代码，否则极易破坏已跑通的逻辑。请严格遵守以下 4 步法：

1. **先改需求文档**：手动（或与 AI 探讨）修改 `specs/...` 文件，将新功能的规则、限制补充进 Goals 或 Non-Goals。
2. **索要图纸更新计划**：打开**全新**的 Composer 窗口，告诉 Agent 需求已更新，强制它先交出**技术更新计划**（例如它打算改哪些文件？数据存哪里？逻辑插在什么生命周期里？）。
3. **审查并动工**：如果你觉得它的设计不够好（比如把长期状态存在了内存变量里），就在窗口里打回重做。直到设计无懈可击，再回复“图纸通过，请执行修改”。
4. **测试与提交**：新功能测试无误后，呼叫 `git-committer` 使用 `[NF]: (功能模块) 说明` 提交代码。
